# The Three-Unit Deployment Model That Changed How I Ship Pipelines

### Stop thinking about one pipeline. Start thinking about three units. Here's why it matters.

![The three-unit deployment model](../diagrams/02-three-unit-model.svg)

---

The first thing most teams do when they're given "move our mainframe data to BigQuery" is draw a diagram.

Mainframe on the left. BigQuery on the right. One big rectangle in the middle labelled `data-pipeline`. An arrow going left-to-right. Sometimes, for ambition, a second arrow going right-to-left.

This is the wrong picture.

It's wrong because it suggests the pipeline is *one thing*, built and deployed as a unit. In reality, a production data pipeline is at least three separate concerns, each with a different owner, different lifecycle, different failure mode.

If you learn nothing else from this whole series, learn this: **stop thinking about one pipeline. Start thinking about three units.**

---

## The three units

Every real pipeline has these three roles:

**Ingestion** — get the data in, in a shape the platform can reason about.
**Transformation** — reshape it into something useful to the business.
**Orchestration** — decide when things run, in what order, and what happens when something breaks.

Lump them together and you have:

- A SQL typo that breaks your Dataflow deploy.
- An orchestration DAG that needs updating every time a new table appears.
- A delayed dashboard where nobody can tell whether it's the fault of the mainframe, the parser, the transform, or the scheduler.

Split them apart and every one of those problems disappears.

---

## Unit 1: Ingestion

Job: read files from a GCS landing bucket, validate them, and load the records into the ODP — Original Data Product — in BigQuery. One table per entity, untransformed, audit-columned, ready for reference.

In `gcp-pipeline-framework`, this is a Dataflow Flex Template. A Docker image, registered with Dataflow, launched on demand with a parameter set.

A few principles that make it work:

- **One entity per run.** Each Dataflow job handles one logical entity's files. Simple code, focused tests, tight failure domain. Customers failing can't break accounts.
- **Schema-driven.** An `EntitySchema` describes each entity. Validation, BigQuery table creation, PII detection all read from that one definition. Change the schema — behaviour changes.
- **HDR/TRL aware.** Mainframe files have headers and trailers. The framework parses both and refuses to load a file whose trailer says 500,000 rows when the body has 499,998.
- **Split-file reassembly.** If the mainframe sends `customers.part01`, `customers.part02`, etc., the framework joins them transparently.
- **Error quarantine.** Bad records don't fail the job. They're routed to an error bucket with enough metadata to replay later.

Result: a black box with a narrow contract. Files in, ODP rows out, errors isolated. You could swap the implementation for Go tomorrow and the rest of the pipeline wouldn't notice.

## Unit 2: Transformation

Job: read from the ODP, produce the FDP — Foundation Data Product. The clean, business-shaped, analytics-ready layer.

Implementation: dbt. Models organised like this:

```
models/
├── staging/        # 1:1 with ODP, light cleanup
├── fdp/            # JOIN and MAP models — the business layer
├── marts/          # Domain-specific mart views
└── analytics/      # Ad-hoc views and aggregations
```

Two transformation patterns, always. I'll write a full post on this, but the short version:

- **JOIN pattern** — some FDP tables need data from multiple ODP tables. Example: `event_transaction_excess` joins customers and accounts. The transform must wait for both loads.
- **MAP pattern** — other FDP tables map one-to-one from a single ODP table. They can transform immediately.

Most real pipelines need both. Most frameworks pretend they don't. This one makes the distinction explicit.

Two framework macros do heavy lifting:
- `generate_audit_columns()` injects lineage (`run_id`, `source_system`, `load_timestamp`).
- `apply_pii_masking()` masks fields marked PII in the `EntitySchema`.

Get them once, apply everywhere. No drift.

## Unit 3: Orchestration

Job: decide when each of the above runs, in what order, and what to do when something breaks.

Implementation: Apache Airflow on Cloud Composer. Five DAGs:

- `pubsub_trigger` — listens for `.ok` files on Pub/Sub, fires the right ingestion DAG.
- `ingestion` — launches the Dataflow Flex Template.
- `transformation` — runs dbt, gated on JOIN preconditions.
- `error_handling` — manages the quarantine lifecycle.
- `status` — writes consolidated run state for dashboards.

The magic is that **none of these DAGs are hand-written**. They're all generated by a factory from a YAML config:

```yaml
system_id: generic
entities:
  customers:
    schedule: "@daily"
    pattern: JOIN
    fdp_models: [event_transaction_excess]
  accounts:
    schedule: "@daily"
    pattern: JOIN
    fdp_models: [event_transaction_excess]
  decision:
    schedule: "@daily"
    pattern: MAP
    fdp_models: [portfolio_account_excess]
```

Adding a new entity is a four-line YAML change. No DAG edits.

A note on cost: Cloud Composer starts at about $300/month before you schedule a single task. The framework treats Composer as **opt-in** — the default deploy doesn't provision it. Smaller teams substitute Cloud Functions + Cloud Run Jobs and save the $300. More on this in a later post.

## Why this model scales

Three reasons.

**Unit isolation.** Teams can own different units without stepping on each other. An SQL analyst modifies the transformation layer without touching Beam code. A platform engineer upgrades orchestration without risking transformation logic. The blast radius of any change is one unit.

**Lifecycle independence.** Ingestion changes when mainframe extracts change. Transformation changes when business logic changes. Orchestration changes when operational policy changes. Different release cadences. Different CI path filters. Different review processes.

**Reuse by substitution.** Because the contract between units is narrow (ODP table, FDP table, Pub/Sub event), you can swap any single unit with an alternative. Dataflow too expensive for a tiny entity? Replace that ingestion with a Cloud Function. Transformation and orchestration don't care.

## What a successful run looks like, end to end

Since this is the architectural chapter, let me walk you through one full run:

1. The mainframe drops `customers.csv` and `customers.ok` into the GCS landing bucket.
2. GCS publishes a notification to Pub/Sub.
3. `pubsub_trigger` DAG wakes up, matches the entity, fires `ingestion_customers`.
4. `ingestion_customers` launches a Dataflow job from the Flex Template.
5. Dataflow reads, parses HDR/TRL, validates against `CustomerSchema`, writes valid rows to `odp_generic.customers`, routes rejects to the error bucket.
6. The ingestion DAG writes a row to `job_control.pipeline_runs`.
7. `transformation` DAG checks `EntityDependencyChecker`. For MAP entities it fires immediately. For JOIN entities it waits for the full source set.
8. dbt runs. FDP tables rebuild with audit columns and PII masking.
9. `status` DAG writes the consolidated record.
10. Dashboards, downstream systems, and `fdp-trigger` hear about the new data.

Every step is observable. Every step is cost-attributed. Every step is testable in isolation. Every step can be replayed from checkpoint.

That's the whole point.

---

## The biggest mistake I see

Teams collapse these units into one for "simplicity." Six months later they have a monolith where every change requires regression testing the whole pipeline. Adding a new entity takes two weeks. On-call is a nightmare. Nobody knows who owns what.

Split the units. Three CI paths. Three deploy targets. Three review processes. Three sets of tests. Your future self will thank you.

## Try it

```bash
pip install gcp-pipeline-framework
python -m gcp_pipeline_framework.reconstruct --dest ~/my-pipeline
cd ~/my-pipeline
ls deployments/
# original-data-to-bigqueryload   <- Unit 1: Ingestion
# bigquery-to-mapped-product      <- Unit 2: Transformation
# data-pipeline-orchestrator      <- Unit 3: Orchestration
```

Three units. Visible in the filesystem. No mystery.
