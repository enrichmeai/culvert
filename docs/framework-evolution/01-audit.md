# Framework Cloud-Coupling Audit

A read-only inventory of `gcp-pipeline-libraries/` against the goal of repositioning the codebase as a cloud-neutral pipeline framework with GCP/AWS/Azure as swappable modules ("Spring for data pipelines"). Findings are based on source inspection only — no code was modified.

## 1. Executive summary

The framework is more cloud-neutral than its name suggests, but less neutral than its module names suggest. Of roughly 25,000 lines of library Python across six packages, my rough estimate is that **45–55% is already cloud-neutral in spirit** (pure-Python data quality, error taxonomy, schema modelling, run-id generation, structured logging, quarantine bookkeeping, observability primitives, dbt SQL macros, test scaffolding), another **15–20% is cloud-neutral logic with a thin GCP wrapper around it** that pluggable-backend refactors can split cleanly (error storage, recovery checkpoints, alert backends, OTEL exporters), and the remaining **25–30% is genuinely GCP-coupled** by design (Beam-on-Dataflow IO, BigQuery cost estimation constants, Composer/Airflow DAG factory, GCS-specific clients).

The headline finding is that the heavy `gcp_pipeline_core` package is mis-labelled. Three of its subpackages — `data_quality`, `data_deletion`, `error_handling` (excluding `storage.py`'s GCS class), large parts of `audit`, all of `utilities.run_id`, all of `utilities.logging`, all of `monitoring.metrics`/`monitoring.health`/`monitoring.types`/`monitoring.observability`, `monitoring.otel.tracing`/`context`, and `file_management.hdr_trl` — have **zero references to `google.cloud`, BigQuery, GCS, Dataflow, or Pub/Sub**. They are coherent cloud-neutral domains that happen to live in a package called `gcp_pipeline_core`. A rename and a small number of file moves would shift these to a real `pipeline-core` without touching their public behaviour. The framework already practices pluggable-backend patterns in three places (`ErrorStorageBackend`, `AlertBackend`, `OTELExporterType`), which signals that the team has the right design instincts; those patterns just need to be applied more widely and the GCP implementations relocated.

What is genuinely GCP-coupled and should stay coupled: `clients/bigquery_client.py`, `clients/gcs_client.py`, `clients/pubsub_client.py`, `job_control/repository.py` (BigQuery SQL embedded throughout), `finops/tracker.py` (BigQuery/GCS/Pub/Sub pricing constants and client objects), `audit/publisher.py` (Pub/Sub), the GCS-backed implementations of `recovery.py` and `storage.py`, the entire `gcp-pipeline-beam` package (Apache Beam pipelines authored for Dataflow runner), and the entire `gcp-pipeline-orchestration` package (Airflow DAG factory built around `airflow.providers.google.cloud.*`). The dbt macros in `gcp-pipeline-transform` are also GCP-coupled in practice — `add_audit_columns` uses BigQuery syntax (`current_timestamp()` is fine but `ALTER TABLE … ADD COLUMN IF NOT EXISTS` and `RPAD/SUBSTRING` semantics differ across warehouses).

The refactor cost is moderate but not trivial. The bulk of the work is mechanical: renaming Python packages (every `gcp_pipeline_*` becomes either `pipeline_*` or `pipeline_gcp_*`), updating ~150 import statements across libraries, and moving roughly 25 GCP-coupled files out of the core into a new `pipeline-gcp` package. The hard part is choosing the boundary contracts (Source, Sink, Warehouse, Orchestrator, BlobStore) and pushing them upstream into the core, because today the Beam pipelines, the job-control repository, and the DAG factory all couple directly to concrete GCP types without going through abstractions. That's a real refactor — call it 2–4 weeks of focused work — but it is achievable without changing the framework's external behaviour or breaking the deployment story.

The strategic claim "Spring for data pipelines" is defensible if you scope it precisely: the framework's domain primitives (run lifecycle, audit, reconciliation, data quality, error classification, observability) are cloud-neutral and can be packaged as such. Its execution primitives (Beam-on-Dataflow, Airflow-on-Composer, BigQuery as warehouse) are GCP-specific and should be packaged as such. Selling the former as the framework, with the latter as the first reference implementation, is honest. Selling the entire repo as cloud-neutral would not be.

## 2. Per-package audit

### 2.1 `gcp-pipeline-core/`

**What it does.** This is the umbrella library for framework primitives consumed by every other package. It contains nine subpackages: `clients` (GCP service clients), `audit` (lineage, reconciliation, trail, publishing), `data_quality` (dimension checkers, scoring, anomaly detection), `data_deletion` (malformation detection, quarantine, deletion, recovery checkpoints), `error_handling` (typed exceptions, classifier, storage, retry policy), `finops` (cost tracking and labelling), `job_control` (pipeline-job state machine and repository), `file_management.hdr_trl` (header/trailer record parsing), `monitoring` (metrics, health, alerts, OTEL bridge), and `utilities` (run IDs, structured logging, GCS path helpers). Total: 63 Python files, ~8,600 lines.

**GCP coupling found.** The hard-coupling is concentrated in a handful of files. `clients/bigquery_client.py`, `clients/gcs_client.py`, and `clients/pubsub_client.py` import `google.cloud.bigquery`, `google.cloud.storage`, and `google.cloud.pubsub` at module level — these are pure GCP service wrappers and don't pretend otherwise. `job_control/repository.py` is the deepest coupling in core: every method on `JobControlRepository` builds parameterised BigQuery SQL and executes it via `bigquery.Client.query()` (49 BigQuery references across 511 lines). `finops/tracker.py` imports `bigquery`, `storage`, and `pubsub_v1` at module level and has BigQuery-specific cost constants (`BQ_COST_PER_TIB = 6.25`); the cost trackers take concrete `bigquery.QueryJob` and `storage.Blob` arguments. `audit/reconciliation.py` has a `reconcile_with_bigquery` method that imports `google.cloud.bigquery` lazily and queries a destination table. `audit/publisher.py` constructs a `PubSubClient` and publishes JSON-serialised audit records to a Pub/Sub topic. `error_handling/storage.py` defines an abstract `ErrorStorageBackend` and an `InMemoryErrorStorage` alongside a `GCSErrorStorage` that does `from google.cloud import storage` inside its methods. `data_deletion/recovery.py` defines an in-memory `RecoveryManager` and a `GCSRecoveryManager` subclass that persists checkpoints to GCS. `monitoring/alerts.py` has a `CloudMonitoringBackend` for Google Cloud Monitoring alongside `DatadogAlertBackend`, `SlackAlertBackend`, and a `LoggingAlertBackend`. `monitoring/otel/config.py` and `monitoring/otel/provider.py` understand multiple OTEL exporters including `gcp_otlp` and `gcp_trace` and a `gcp_project_id` field. `utilities/gcs_discovery.py` builds `gs://` paths and discovers files via a GCS client interface. `file_management/hdr_trl/parser.py` has a single GCS read path (line 169–178) where, if a `gs://` URI is passed, it falls back to instantiating `gcp_pipeline_core.clients.GCSClient`.

**Cloud-neutral content.** The following files contain zero references to `google`, `gcs`, `bigquery`, or any other cloud-specific term: every file under `data_quality/` (`checker.py`, `dimensions.py`, `scoring.py`, `anomaly.py`, `reporting.py`, `types.py` — pure record-list logic over `List[Dict[str, Any]]`); every file under `data_deletion/` *except* `recovery.py` (i.e. `detector.py`, `quarantine.py`, `deletion.py`, `types.py`, `framework.py`); every file under `error_handling/` *except* `storage.py` (the entire exception hierarchy in `errors.py`, the `ErrorHandler` and `ErrorClassifier` in `handler.py`, the `ErrorContext` decorator in `context.py`, the `PipelineError` dataclass and `ErrorConfig` in `models.py`, and `types.py`); `audit/records.py`, `audit/trail.py`, `audit/lineage.py` (the lineage tracker just builds a dict); `finops/models.py` (the `CostMetrics` dataclass — fields like `billed_bytes_scanned`, `slot_millis` happen to map cleanly to GCP but the type itself is generic) and `finops/labels.py` (key/value labels — uses lowercase `system`/`environment`/`managed_by`, which is GCP-friendly but not GCP-specific); `job_control/models.py` and `job_control/types.py` (the `PipelineJob` dataclass and `JobStatus`/`FailureStage`/`JobType` enums); `file_management/hdr_trl/types.py`, `constants.py`, and most of `parser.py` (the parser logic is regex over text lines); `utilities/run_id.py` (pure UUID + timestamp); `utilities/logging.py` (a custom `StructuredJsonFormatter` and `StructuredLogger` — comments mention Cloud Logging but the code emits plain JSON to a stream); `monitoring/metrics.py`, `monitoring/health.py`, `monitoring/types.py`, `monitoring/observability.py` (thread-safe in-memory counters/gauges); `monitoring/otel/tracing.py`, `monitoring/otel/context.py`, `monitoring/otel/metrics_bridge.py` (standard OTEL API usage); and `schema.py` (`SchemaField`/`EntitySchema` dataclasses — there is a `to_bq_schema()` convenience method, but it's optional and easily renamed/extended).

**Verdict.** Mixed — needs splitting. Roughly 70% of this package is cloud-neutral in fact; the rest is a small, well-isolated set of GCP-specific files that are already in the right rough shape (pluggable backends in three places) and can be lifted out cleanly.

**Refactor list.**
1. Rename `gcp_pipeline_core` → `pipeline_core`. Move the GCP-bound files into a new `pipeline_gcp` package.
2. Move `clients/bigquery_client.py`, `clients/gcs_client.py`, `clients/pubsub_client.py` to `pipeline_gcp/clients/`. They're cohesive and unambiguous.
3. Move `job_control/repository.py` to `pipeline_gcp/job_control/bigquery_repository.py`. Extract a `JobControlRepository` protocol from its current public surface (the `create_job`, `update_status`, `mark_failed`, `get_job`, `get_pending_jobs`, `mark_retrying`, `cleanup_partial_load`, `get_failed_jobs`, `get_fdp_job_status`, `update_cost_metrics` methods) and keep that protocol in `pipeline_core/job_control/`. The `PipelineJob` dataclass and the enums stay in core.
4. Move `finops/tracker.py` to `pipeline_gcp/finops/tracker.py`. Keep `CostMetrics` and `FinOpsLabels` in `pipeline_core/finops/`; the labels class needs only renaming the default `project` value away from `gcp-pipeline-framework` and the `to_dict` docstring away from "GCP-compatible".
5. Move `audit/publisher.py` (the Pub/Sub publisher) to `pipeline_gcp/audit/pubsub_publisher.py`. Define an `AuditEventPublisher` protocol in `pipeline_core/audit/` with a single `publish(record)` method.
6. Move `audit/reconciliation.py`'s `reconcile_with_bigquery` method (and the lazy `from google.cloud import bigquery` import) to a `BigQueryReconciliationEngine` subclass in `pipeline_gcp/audit/`. The base `ReconciliationEngine` and its `reconcile_counts` method are pure arithmetic and stay in core.
7. Move `error_handling/storage.py`'s `GCSErrorStorage` class to `pipeline_gcp/error_handling/gcs_storage.py`. The `ErrorStorageBackend` ABC and `InMemoryErrorStorage` stay in core. This is already the architecture — it just needs the package split.
8. Move `data_deletion/recovery.py`'s `GCSRecoveryManager` to `pipeline_gcp/data_deletion/gcs_recovery.py`. `RecoveryManager` and `RecoveryPoint` stay in core. Same story.
9. Move `monitoring/alerts.py`'s `CloudMonitoringBackend` to `pipeline_gcp/monitoring/cloud_monitoring_alerts.py`. `AlertBackend`, `AlertManager`, `LoggingAlertBackend`, `DatadogAlertBackend`, `SlackAlertBackend` all stay in core (Datadog and Slack are already cloud-neutral integrations).
10. Move `utilities/gcs_discovery.py` to `pipeline_gcp/utilities/gcs_discovery.py`. Add a `pipeline_core/utilities/blob_discovery.py` with a `BlobStore` protocol (methods like `list_prefix`, `read`, `write`, `exists`) so the file-discovery functions can be re-expressed against the protocol.
11. Rename `monitoring/otel/config.py`'s class names and exporter values away from `gcp_` prefixes where the exporter itself isn't actually GCP-specific. The `GCP_OTLP` and `GCP_TRACE` enum members can stay — they are genuinely GCP-targeted exporters.
12. In `file_management/hdr_trl/parser.py`, replace the `gs://`-sniffing block (lines 169–179) with a `BlobStore` dependency injection. Default behaviour stays the same for local files.

### 2.2 `gcp-pipeline-beam/`

**What it does.** Apache Beam pipeline scaffolding intended for the Dataflow runner. Provides a `BasePipeline` base class, GCP-aware pipeline options, a builder for assembling stages, Beam `DoFn`s for BigQuery and GCS I/O, Pub/Sub publishers, CDC streaming, and a transforms library (CSV parsing, deduplication, enrichment, filtering, parsers, PII masking, transformers, validators, windowing). Also includes file-management code for archiving and lifecycle management of GCS objects, and a validators package with composable record-level validators. Total: 45 Python files, ~8,400 lines.

**GCP coupling found.** This package is GCP-coupled by construction. `pipelines/base/pipeline.py` imports `apache_beam`; `pipelines/base/options.py` defines `GCPPipelineOptions` with `--gcp_project`, `--gcp_autoscaling_algorithm`, `--gcp_num_workers`, `--gcp_max_num_workers` parameters. `pipelines/beam/io/bigquery.py` defines `WriteToBigQueryDoFn` and other BigQuery writers. `pipelines/beam/io/bigquery_retry.py` is BigQuery-specific retry logic. `pipelines/beam/io/gcs.py` imports `apache_beam.io.gcp.gcsio`. `pipelines/beam/pubsub/publishers.py` imports `google.cloud.pubsub_v1`. `pipelines/beam/resource_config.py` and `pipelines/beam/builder.py` import GCP libraries. `pipelines/beam/streaming/cdc.py` reads from Pub/Sub. The `file_management/` subpackage (`archiver.py`, `lifecycle.py`, `metadata.py`) all use `google.cloud.storage`. About 12 of 45 files import `google.cloud.*` directly; almost all of the rest import Apache Beam.

**Cloud-neutral content.** A meaningful minority. `pipelines/beam/transforms/csv_parser.py` is a robust CSV parser with a `CSVErrorType` enum and recovery strategies — the logic is pure Python over text rows; it inherits from `beam.DoFn` but the parsing logic is portable. `pipelines/beam/transforms/pii.py`, `deduplicators.py`, `enrichers.py`, `filters.py`, `transformers.py`, `windowing.py` use Beam types in signatures but the transformation logic is largely warehouse-agnostic. The `validators/` subpackage (`classes.py`, `code.py`, `date.py`, `generic.py`, `numeric.py`, `schema_validator.py`, `types.py`) is composable record-level validation — has a small Beam touch but the validators themselves know nothing about clouds. `file_management/split_file_handler.py`, `integrity.py`, `policy.py`, `types.py`, `validator.py` carry portable file-handling logic.

**Verdict.** Cloud-specific module — keep, rename to `pipeline-gcp-beam` (or `pipeline-runner-dataflow`). The package is Apache Beam authored for the Dataflow runner; lifting it out of the GCP family wouldn't be intellectually honest and would invite users to deploy Beam pipelines they expect to run elsewhere. Two specific subdirectories — `validators/` and the warehouse-agnostic `transforms/` — are good candidates for extraction into a cloud-neutral `pipeline-validators` and `pipeline-transforms` package, but that's optional polish, not a structural requirement.

**Refactor list.**
1. Rename `gcp_pipeline_beam` → `pipeline_gcp_beam`. Keep all of `pipelines/beam/` and `pipelines/base/` here — they all transitively assume Dataflow as the execution environment.
2. Extract `validators/` into a standalone `pipeline-validators` package. It depends on nothing cloud-specific; classes like `RequiredValidator`, `RegexValidator`, `SchemaValidator`, `validate_numeric_range`, `validate_date` are pure logic.
3. Consider extracting the warehouse-neutral parts of `transforms/` (csv_parser, pii, deduplicators, enrichers, filters, transformers, validators, windowing) into `pipeline-beam-transforms` — these are Beam DoFns but not Dataflow-specific in their logic.
4. Rename `GCPPipelineOptions` → `DataflowPipelineOptions` in `pipelines/base/options.py`. The class is genuinely Dataflow-specific (autoscaling algorithm, worker counts), so the rename clarifies intent.
5. The `file_management/` subpackage here (archiver, lifecycle, metadata) is GCS-specific. Either leave it in `pipeline_gcp_beam/file_management/` or unify it with the `hdr_trl` code into a `pipeline_gcp/file_management/` namespace. Recommend the latter for consistency.
6. Replace the `BasePipeline.__init__`'s direct imports from `gcp_pipeline_core.{audit,error_handling,monitoring,utilities}` with the renamed `pipeline_core.*` imports, plus a fallback to `pipeline_gcp.*` for the GCP-specific repository.

### 2.3 `gcp-pipeline-framework/`

**What it does.** A thin umbrella distribution. It has exactly one Python module (`__init__.py`) and a `cli.py`. The package's actual purpose is to bundle non-Python assets (docs, infrastructure Terraform, GitHub Actions workflows, deployment Dockerfiles, test data) and provide an `export_project()` function that recreates the full reference-repo layout in a target directory. Total: 3 Python files, ~150 lines.

**GCP coupling found.** None at the Python level. There are zero imports of `google.cloud`, no references to BigQuery, GCS, Dataflow, Composer, Airflow, or Pub/Sub in the Python code. The coupling is indirect: the *contents* of the bundled `infrastructure/` directory are Terraform for GCP, the bundled `workflows/` are GitHub Actions that deploy to GCP, and the bundled `deployments/` are GCP-specific Dockerfiles and cloudbuild configs.

**Cloud-neutral content.** The Python is entirely cloud-neutral — `_package_root()`, `get_docs_path()`, `get_infrastructure_path()`, `get_workflows_path()`, `get_config_path()`, `get_deployments_path()`, `get_test_data_path()`, `list_docs()`, `export_project()`, `_copy_tree()`. It's a glorified asset bundler.

**Verdict.** Core-eligible as-is — the Python code, that is. The bundled assets are GCP-specific and should be moved to a sibling distribution.

**Refactor list.**
1. Rename `gcp_pipeline_framework` → `pipeline_framework`.
2. Keep the Python in `pipeline-framework`. It is genuinely cloud-neutral plumbing.
3. Move the bundled `infrastructure/`, `workflows/`, `deployments/` resources into a new `pipeline-gcp-deployment` package (or just leave them in the reference repo and stop bundling them into a Python wheel — they don't belong inside `site-packages`).
4. Generalise `export_project()` to accept a target distribution name so consumers could call `export_project(dest="my-project", cloud="gcp")` and have it pull from the appropriate cloud-specific assets package.

### 2.4 `gcp-pipeline-orchestration/`

**What it does.** A config-driven Airflow DAG factory. Reads a `system.yaml` configuration and generates four DAGs per system: a Pub/Sub trigger DAG, an ingestion DAG that runs Dataflow, a transformation DAG that runs dbt, and a status DAG that alerts on incomplete pipelines. Also provides reusable Airflow operators (`BaseDataflowOperator`), sensors (`BasePubSubPullSensor`, `DataflowJobSensor`), callbacks (DLQ publishing, file quarantine, validation/routing failure handlers), an `EntityDependencyChecker`, and a Secret Manager hook. Total: 25 Python files, ~5,200 lines.

**GCP coupling found.** Substantial and structural. `factories/dag_factory.py` imports from `airflow.providers.standard.operators` and from `gcp_pipeline_core.job_control` and `gcp_pipeline_core.error_handling`. It is fundamentally an Airflow DAG factory. `operators/dataflow.py` wraps `DataflowTemplatedJobStartOperator`. `sensors/pubsub.py` extends `airflow.providers.google.cloud.sensors.pubsub.PubSubPullSensor`. `sensors/dataflow.py` and `hooks/secrets.py` import from `airflow.providers.google.cloud.*` and `google.cloud.secretmanager`. `callbacks/quarantine.py` and `callbacks/dlq.py` assume Pub/Sub for DLQ and GCS for quarantine. The `routing/` and `factories/config.py` modules are config dataclasses that happen to be cloud-neutral but are wired into the GCP-coupled DAG factory.

**Cloud-neutral content.** A small island. `factories/config.py` (`RetryPolicy`, `TimeoutConfig`, `ScheduleConfig`, `DefaultArgs`, `TaskConfig`, `DAGConfig` dataclasses) is generic — these are scheduler-agnostic dataclasses about retries, timeouts, schedules, and task configuration. `factories/validators.py` validates those dataclasses with no cloud awareness. `routing/router.py`, `routing/config.py`, `routing/yaml_selector.py` are file-type routing logic. `callbacks/types.py` and `callbacks/handlers.py` define error-type enums and handler protocols; the handlers themselves get GCP-coupled through `dlq.py` and `quarantine.py`, but the abstractions don't have to be.

**Verdict.** Cloud-specific module — keep, rename to `pipeline-gcp-composer` (or `pipeline-orchestration-airflow-gcp`). The DAG factory is built for Composer/Airflow with the Google Cloud provider; that's its entire reason to exist. Extracting a thinner cloud-neutral routing layer is possible but is a separate piece of work.

**Refactor list.**
1. Rename `gcp_pipeline_orchestration` → `pipeline_gcp_composer` (or similar — make the Composer dependency explicit in the name).
2. Extract `factories/config.py` and `factories/validators.py` into a new `pipeline-core/orchestration/` namespace. These dataclasses are scheduler-agnostic and would be reused by an AWS Step Functions or Azure Data Factory adapter.
3. Extract `routing/` (`router.py`, `config.py`, `yaml_selector.py`) into `pipeline-core/routing/`. File-type-to-pipeline-config routing is a generic concern.
4. Keep `operators/`, `sensors/`, `hooks/`, `callbacks/{dlq,quarantine}.py`, `factories/dag_factory.py`, `factories/_dag_builders.py` in the GCP-specific Composer package.
5. Move `callbacks/types.py` and `callbacks/handlers.py` to core (they declare types and signatures; their GCP implementations live in `dlq.py`/`quarantine.py`).
6. Introduce a thin `Orchestrator` contract in core (methods like `schedule`, `add_task`, `wire_dependency`) that the GCP DAG factory implements. This unlocks future AWS/Azure orchestrator implementations.

### 2.5 `gcp-pipeline-tester/`

**What it does.** Testing utilities. Provides base test classes (`BasePipelineTest`, `BaseValidationTest`, `BaseBeamTest`, `PipelineScenarioTest`), record/pipeline/config builders, assertion helpers, BDD step definitions, mock clients for BigQuery/GCS/Pub/Sub, pytest fixtures for those services, and a `DualRunComparison` utility that compares a source CSV file against a BigQuery target table. Total: 33 Python files, ~3,100 lines.

**GCP coupling found.** Light and mostly cosmetic. The only file with real GCP coupling is `comparison/dual_run.py`, which imports `google.cloud.bigquery` at module top and queries a BigQuery target table. `builders/config_builder.py` has a `with_gcp_project` method and uses `gs://test-bucket/input.csv` as a default `source_file`. `fixtures/gcs.py`, `fixtures/bigquery.py`, `fixtures/common.py`, and `fixtures/__init__.py` reference GCS/BigQuery by name but their bodies are `Mock()` objects and string constants — there are no real `google.cloud.*` imports anywhere in the fixtures except the conftest helpers. `mocks/bigquery_mock.py`, `mocks/gcs_mock.py`, `mocks/pubsub_mock.py` are hand-written mock classes that contain no real Google API imports.

**Cloud-neutral content.** Most of the package. `base/pipeline_test.py`, `base/validation_test.py`, `base/scenario_test.py`, `base/result.py` are generic unittest base classes. `base/beam_test.py` depends on Beam but not on GCP. `assertions/`, `bdd/` step definitions, `builders/record_builder.py`, `builders/pipeline_builder.py` are all generic.

**Verdict.** Core-eligible after rename + light refactor.

**Refactor list.**
1. Rename `gcp_pipeline_tester` → `pipeline_tester`.
2. Move `comparison/dual_run.py` (the only file with real BigQuery imports) to a `pipeline_gcp/testing/dual_run.py` location, or generalise it to accept a `Warehouse` adapter so the comparison can target Snowflake/Redshift/etc.
3. Rename `mocks/bigquery_mock.py` → `mocks/warehouse_mock.py` (rename internal class `BigQueryClientMock` → keep alias, add `WarehouseClientMock`). Same treatment for `gcs_mock.py` → `blob_store_mock.py`.
4. Move `fixtures/bigquery.py`, `fixtures/gcs.py` to `pipeline_gcp/testing/fixtures/` with thin aliases left behind.
5. Remove `with_gcp_project` from `PipelineConfigBuilder` in favour of `with_cloud_project` or just `with_project`. The default `gs://test-bucket/input.csv` source path should become a parameter, not a default.
6. The `BasePipelineTest` docstring still says "GCP pipeline tests" — string-update only.

### 2.6 `gcp-pipeline-transform/`

**What it does.** A dbt macro library. The Python package itself is empty (a single `__init__.py` with a `__version__`); the real content is four SQL files under `src/gcp_pipeline_transform/dbt_shared/macros/`: `audit_columns.sql`, `data_quality_check.sql`, `enrichment.sql`, and `pii_masking.sql`. The README explicitly states "NO Apache Beam or Airflow dependencies." Total: 1 Python file, ~5 lines + ~4 SQL files of macro logic.

**GCP coupling found.** No Python coupling. The SQL macros, however, are written for BigQuery dialects in places: `audit_columns.sql`'s `apply_audit_columns` macro uses `ALTER TABLE … ADD COLUMN IF NOT EXISTS` (BigQuery- and Postgres-compatible but not Redshift); `pii_masking.sql` uses `RPAD`, `SUBSTRING`, `LENGTH`, `CONCAT` which are mostly portable but the negative-index `SUBSTRING({{ column }}, -4)` is BigQuery/Snowflake but not standard SQL. These are addressable with dbt's `adapter` macros but currently they assume BigQuery.

**Cloud-neutral content.** The Python package is cloud-neutral by construction (it has no Python code). The macro library *concept* is cloud-neutral (dbt itself runs against many warehouses). The current macro implementations are written assuming BigQuery dialect.

**Verdict.** Core-eligible after rename + light refactor. The dbt macro library can move to a cloud-neutral name, but the SQL inside should be wrapped in `{% if target.type == 'bigquery' %}` blocks (or moved to BigQuery-specific files) before the rename is honest.

**Refactor list.**
1. Rename `gcp_pipeline_transform` → `pipeline_transform`.
2. Wrap dialect-sensitive macro bodies in `{% if target.type == 'bigquery' %} … {% elif target.type == 'snowflake' %} … {% endif %}` blocks, or move dialect-specific code into `dbt_shared/macros/bigquery/` and add a `dbt_shared/macros/snowflake/` placeholder.
3. The README needs updating to reflect that the macros are warehouse-portable (or to call out which are not).
4. Consider splitting into `pipeline-transform-core` (dialect-neutral macros) and `pipeline-transform-bigquery` (BigQuery dialect implementations).

## 3. Cross-cutting findings

### 3.1 Naming patterns that need normalising

Every Python package is `gcp_pipeline_*`, every distribution is `gcp-pipeline-*`, and the docstrings of nearly every module begin with "GCP Pipeline Framework — …". This is the most pervasive form of in-name-only coupling: about 60 module-level docstrings open with "GCP Pipeline Framework" even when the file contains no GCP code (`data_quality/checker.py`, `error_handling/errors.py`, `audit/lineage.py`, and many more). A simple search-and-replace pass would clear most of this. Class naming is more uneven: `JobControlRepository` is well named (it doesn't say BigQuery anywhere), but `BigQueryCostTracker` and `CloudStorageCostTracker` are correctly GCP-named (because they take GCP types), `GCSErrorStorage`/`GCSRecoveryManager`/`CloudMonitoringBackend` are correctly named (they are GCP implementations), and `GCPPipelineOptions` in Beam should arguably be `DataflowPipelineOptions`. The good news: there is no widespread case of a class called `BigQueryFoo` doing pure-Python work — the team has been disciplined about naming GCP-specific classes for the cloud they target.

### 3.2 Config schema coupling

The framework leaks a GCP project/dataset assumption in three places. First, `JobControlRepository.__init__` takes `project_id`, `dataset`, `table` parameters and builds a `f"{project_id}.{dataset}.{table}"` table identifier — that triple is BigQuery-shaped. Second, `PipelineConfig` in `gcp-pipeline-beam` has `gcp_project_id` and `bigquery_dataset` fields. Third, `GCPPipelineOptions` exposes `--gcp_project`, `--gcp_autoscaling_algorithm`, `--gcp_num_workers`. The `system.yaml` config that drives the orchestration DAG factory has entries like `gcp_project`, `dataflow_template_path`, `dataset` that assume the GCP triplet. A cloud-neutral framework would express this as `{cloud, project_or_account, namespace, resource}` or as opaque resource URIs. None of these are difficult to abstract — they just haven't been.

### 3.3 Auth coupling

The framework assumes Application Default Credentials are available wherever it runs. `bigquery.Client(project=project_id)`, `storage.Client(project=project_id)`, and `pubsub.PublisherClient()` are constructed without explicit credentials, which means GCP ADC is the implicit assumption. `hooks/secrets.py` in orchestration uses Google Cloud Secret Manager via `airflow.providers.google.cloud.hooks.base.CloudBaseHook`. There is no provider abstraction over secret management (a `SecretProvider` protocol would unify GCP Secret Manager, AWS Secrets Manager, Azure Key Vault). This is a moderate hidden coupling — it doesn't appear in many files but it's load-bearing in the orchestration layer.

### 3.4 Tests — coupling and runnability

The test situation is healthier than the production code. There is a clear pattern of using mocks (`mocks/bigquery_mock.py`, `mocks/gcs_mock.py`, `mocks/pubsub_mock.py` in the tester package, plus extensive `unittest.mock` usage throughout `gcp-pipeline-core/tests/`). I sampled `tests/unit/job_control/test_repository.py` and the OTEL tests, and they mock the `google.cloud.bigquery.Client` at the boundary rather than requiring a real BigQuery instance. The `gcp-pipeline-tester` package exists in part to make this pattern available to consumers. The few tests that *do* require GCP credentials are integration-style and easily quarantined. Net assessment: a cloud-neutral core could be tested locally today, no in-memory fake needed.

## 5. Refactor cost estimate

The mechanical cost is large but predictable. Roughly 169 Python files would change name or path: six top-level packages renamed, around 25 files moved from `gcp-pipeline-core` to a new `pipeline-gcp` distribution, and around 5 files moved from `gcp-pipeline-orchestration` into the new core. Every `from gcp_pipeline_core …`, `from gcp_pipeline_beam …`, `from gcp_pipeline_orchestration …`, `from gcp_pipeline_tester …`, and `from gcp_pipeline_transform …` import statement needs updating — I counted approximately 150 such cross-package imports across the libraries plus another ~80 in the reference deployments (`deployments/*/dags/`, `deployments/*/pipelines/`) and around ~30 in the test suites. Most of these are mechanical with `sed`, but some require care because the same name appears in old docstrings, sphinx references, generated egg-info files, and CI scripts. The egg-info directories will need regeneration. The pyproject.toml `name` and `dependencies` fields need updating in every package.

The harder cost is contract extraction. To make the cloud-neutral core actually serve as a contract surface, four protocols need to be designed and pushed into core: `JobControlRepository` (CRUD against a pipeline-job ledger), `BlobStore` (read/write/list/delete object blobs), `Warehouse` (query, load, append rows), and `AuditEventPublisher` (publish audit records). Today these protocols don't exist; the concrete BigQuery/GCS/Pub/Sub classes are referenced directly. Designing those protocols, retrofitting the existing GCP code to implement them, and refactoring `BasePipeline` in Beam and `dag_factory` in orchestration to construct concrete implementations via dependency injection rather than direct import — that is real design work, not mechanical refactoring. I'd estimate 3–5 days of design plus a week of implementation for the protocols alone, plus the mechanical rename pass on top.

Public-API breakage is the bear in the room. Every downstream consumer of `gcp_pipeline_core.audit`, `gcp_pipeline_core.job_control`, etc. would need to update imports. The `gcp_pipeline_framework.export_project()` function bundles assets that may already be referenced by other names. The `system.yaml` schemas consumed by the DAG factory have GCP-shaped fields (`gcp_project`, `dataflow_template_path`). The compatibility-shim option — leave the old `gcp_pipeline_core` package in place as a re-export wrapper around the new `pipeline_core` for one or two release cycles — would let downstream consumers migrate at their own pace, but adds maintenance burden for the duration. Recommend it anyway: a flag-day rename, even with `sed`, will break someone.

The deployment story is the easiest part. The deployment artefacts (Dockerfiles, Cloud Build configs, Terraform) live in `infrastructure/` and `deployments/` and don't reference Python package names except in `requirements.txt`-style files. As long as the new packages publish under both old and new names during a transition (a small `pyproject.toml` tweak), the deploy pipelines keep working.

Tests will mostly survive the rename. Roughly 230 test files live across the libraries; most will need only an import update. The `tests/integration/` style tests that require real GCP credentials are already quarantined behind environment variables and CI gating, so the rename doesn't change their runtime behaviour. Test fixtures that use `gs://` paths can keep using them — the strings are just data — but a future cloud-neutral fixture set would benefit from a `BlobStore`-typed fixture in addition to the GCS-specific one.

Realistic total: I would estimate the rename + protocol design + light pluggability work at 3–5 focused engineer-weeks for a single experienced developer who knows this codebase, plus another week of staging/integration testing in a real GCP environment to confirm nothing regressed. That is not "Spring for data pipelines" as a multi-cloud product — it is, however, enough to honestly position the framework as cloud-neutral at its contract layer, with GCP as the first reference implementation. A genuine multi-cloud product (with AWS and Azure adapters built out and tested) is a much larger investment that would only be worth committing to once the cloud-neutral contracts have proven themselves under the GCP adapter.
